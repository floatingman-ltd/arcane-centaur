== .NET Guide (C# and F#)

Full support for C# and F# including LSP, REPL-driven development, and
format-on-save. Both languages share the .NET SDK and the same iron.nvim
REPL keymaps.

[[quick-start]]
== Quick Start

=== C#

[arabic]
. Open any `.cs` file — Roslyn attaches automatically.
. If the project has multiple `.sln` files, run `:Roslyn target` to
pick the solution.
. Start the REPL with `,sl` on any line, or open it explicitly via
iron.nvim's `:IronRepl`.

=== F#

[arabic]
. Verify prerequisites:
+
[source,sh]
----
dotnet --version
dotnet tool list -g | grep fsautocomplete
----
. Open a `.fs` source file — plugins and LSP attach automatically.
. Start the REPL with `,sl` on any line.

[[csharp-lsp]]
== C# — LSP (Roslyn)

All standard LSP keybindings apply in `.cs` buffers:

[width="100%",cols="50%,50%",options="header",]
|===
|Keys |Action
|`gd` |Go to definition
|`K` |Hover documentation
|`gr` |Find references
|`++<++leader++>++rn` |Rename symbol
|`++<++leader++>++ca` |Code action (includes _Fix all_, _Nested actions_, _Complex edits_)
|`++<++leader++>++e` |Show diagnostic float
|`++[++d` / `++]++d` |Previous / next diagnostic
|===

Roslyn also supports:

* *Multiple solution files* — use `:Roslyn target` to switch between them.
* *Source-generated files* — navigates into generated code transparently.
* *Broad root detection* — set `broad++_++search = true` in the plugin opts if your `.sln` lives above the project root.

---

[[csharp-repl]]
== C# — REPL (csharprepl via iron.nvim)

LocalLeader is `,` in `.cs` buffers.

[cols=",",options="header",]
|===
|Keys |Action
|`,sl` |Send current line to REPL
|`,sc` |Send motion or visual selection
|`,sp` |Send current paragraph
|`,sf` |Send entire file
|`,s++<++CR++>++` |Send carriage return
|`,si` |Interrupt REPL
|`,sq` |Quit / exit REPL
|`,cl` |Clear REPL output
|===

The REPL opens as a horizontal split (40% height) at the bottom of the window.

[[csharp-formatting]]
== C# — Formatting (CSharpier)

Format-on-save is enabled automatically in `.cs` buffers via `conform.nvim`. Manual trigger:

[cols=",",options="header",]

|Keys |Action
|`++<++leader++>++f` |Format buffer or visual selection
|===

CSharpier is opinionated and zero-config — no project-level configuration file required.

[[debugging]]
== Debugging (nvim-dap + netcoredbg)

Breakpoint debugging for C# and F# is provided by https://github.com/mfussenegger/nvim-dap[nvim-dap] +
https://github.com/rcarriga/nvim-dap-ui[nvim-dap-ui] with the https://github.com/Samsung/netcoredbg[netcoredbg]
adapter, which easy-dotnet auto-registers with nvim-dap once both are loaded (there is no
hand-written `dap.adapters` entry — a session therefore only starts from a `.cs`/`.fsharp` buffer,
where easy-dotnet is loaded).

=== Prerequisites

`netcoredbg` is *not* a NuGet / `dotnet tool` package (there is no `dotnet tool install -g netcoredbg`).
Download the prebuilt binary from the
https://github.com/Samsung/netcoredbg/releases[Samsung/netcoredbg releases] and put it on your `$PATH`.

*Linux / WSL2 (x64):*

[source,sh]
----
# Resolve the latest release tag, download the linux-amd64 build, and extract it
NCDBG_VER=$(curl -s https://api.github.com/repos/Samsung/netcoredbg/releases/latest \
  | grep '"tag_name"' | cut -d'"' -f4)
curl -L "https://github.com/Samsung/netcoredbg/releases/download/${NCDBG_VER}/netcoredbg-linux-amd64.tar.gz" \
  -o /tmp/netcoredbg.tar.gz
mkdir -p ~/.local/share/netcoredbg
tar -xzf /tmp/netcoredbg.tar.gz -C ~/.local/share/netcoredbg/

# Add to your shell rc (~/.bashrc or ~/.zshrc), then reload:
echo 'export PATH="$PATH:$HOME/.local/share/netcoredbg"' >> ~/.bashrc
source ~/.bashrc
----

Verify it is on your `$PATH`:

[source,sh]
----
netcoredbg --version
----

[NOTE]
====
If `netcoredbg --version` is not found afterward, the tarball extracted into a nested
`netcoredbg/` subdirectory — point `PATH` at `~/.local/share/netcoredbg/netcoredbg` instead.
====

=== Debug Keymaps

[cols=",,",options="header",]
|===
|Keys |Mode |Action
|`<F9>` / `++<++leader++>++bb` |Normal |Toggle breakpoint
|`<F5>` / `++<++leader++>++bc` |Normal |Continue / start debug session
|`<F10>` / `++<++leader++>++bv` |Normal |Step over
|`<F11>` / `++<++leader++>++bi` |Normal |Step into
|`<F12>` / `++<++leader++>++bo` |Normal |Step out
|`<S-F5>` / `++<++leader++>++bt` |Normal |Terminate debug session
|`++<++leader++>++bu` |Normal |Toggle debug UI
|`++<++leader++>++br` |Normal |Open debug REPL
|===

[NOTE]
====
Every action has both a function-key and a `++<++leader++>++b` binding. The F-keys are the
universal debugger idiom and work on server/SSH terminals; the `++<++leader++>++b` mirrors are for
GUI terminal emulators that intercept function keys before Neovim sees them (commonly `<F11>` for
fullscreen, `<F10>` for a menu). Both are always active — use whichever your terminal passes through.
====

The debug UI opens automatically when a session starts and closes on exit.
Launch configurations are auto-discovered by easy-dotnet from the project structure
(no hand-authored `launch.json` required).

== Testing (easy-dotnet)

https://github.com/GustavEikaas/easy-dotnet.nvim[easy-dotnet.nvim] provides a
.NET test runner, project picker, and solution management using fzf-lua. It does
NOT manage the Roslyn LSP — roslyn.nvim remains the sole C# language server.

[IMPORTANT]
====
easy-dotnet.nvim is a thin client over a separate **`EasyDotnet` server tool**, which powers *all*
its features (debug, test, run, build). Install it once:

[source,sh]

dotnet tool install -g EasyDotnet     # provides the `dotnet-easydotnet` executable

Without it, every `:Dotnet …` action (and `<F5>` debug) errors
`'dotnet-easydotnet' is not executable`. Run `:checkhealth easy-dotnet` to confirm the server and
the other dependencies (dap, fzf-lua, the `c_sharp` parser).

== Typical Workflow

....
 +------------------------------------------------------+
 |  Neovim                                              |
 |  +--------------------------+---------------------+  |
 |  |  Program.cs / hello.fs   |  REPL               |  |
 |  |                          |                     |  |
 |  |  var x = 42;             |  > var x = 42;      |  |
 |  |  Console.WriteLine(x);   |  42                 |  |
 |  |                          |                     |  |
 |  |  ,sl to send line -------+                     |  |
 |  +--------------------------+---------------------+  |
 +------------------------------------------------------+
....

[arabic]
. Open a `.cs` or `.fs` file — LSP starts and diagnostics appear.
. Send snippets to the REPL with `,sl` / `,sc` to explore APIs interactively.
. `++<++leader++>++f` (or save) formats with CSharpier / fsautocomplete.
. Use `gd` / `gr` / `++<++leader++>++ca` for full IDE-style navigation and refactoring.

== Troubleshooting

=== Roslyn does not attach

Check that `Microsoft.CodeAnalysis.LanguageServer` is on your `$PATH`:

[source,sh]
----
which Microsoft.CodeAnalysis.LanguageServer
----

If not found, re-run the install steps in xref:#prerequisites[Prerequisites].

=== fsautocomplete does not attach

Check that it's installed:

[source,sh]
----
dotnet tool list -g | grep fsautocomplete
----

Run `:lua =vim.lsp.get_clients({bufnr=0})` inside Neovim to check whether
the server is attached to the current buffer. To open the LSP log:

[source,vim]
----
:lua vim.cmd('e ' .. vim.lsp.get_log_path())
----

`:checkhealth lsp` gives a broader overview of LSP status.

=== REPL does not open

Check that the REPL backend binary is on your `$PATH`:

[source,sh]
----
which csharprepl      # C#
dotnet fsi --version  # F#
----

[[setup]]
== Setup

=== Installing .NET Global Tools

[source,sh]
----
dotnet tool install -g csharpier    # C# formatter
dotnet tool install -g csharprepl   # C# interactive REPL
dotnet tool install -g fsautocomplete  # F# language server
----

=== Installing the Roslyn Language Server

The Roslyn LSP is the official Microsoft C# language server used in
VS Code's C# Dev Kit. It is *not* installed via `dotnet tool install`,
and it is *not* on the public nuget.org feed. It is published to
Microsoft's Azure DevOps *vs-impl* feed, and every release is a
prerelease (there is no stable `5.x` — the newest is `5.4.0-2.26179.14`
at the time of writing; there is no `5.5`/`5.6`).

*Linux / WSL2 (x64):*

[source,sh]
----
# 1. Resolve the newest version from the Azure DevOps vs-impl feed
PKG=microsoft.codeanalysis.languageserver.linux-x64
FEED="https://pkgs.dev.azure.com/azure-public/vside/_packaging/vs-impl/nuget/v3/flat2/${PKG}"
VERSION=$(curl -s "${FEED}/index.json" | tr ',' '\n' | grep -oE '[0-9][0-9.]+-[0-9.]+' | sort -V | tail -1)
echo "Installing Roslyn LSP ${VERSION}"

# 2. Download the linux-x64 package (self-contained native binary)
mkdir -p ~/.local/share/roslyn
curl -L "${FEED}/${VERSION}/${PKG}.${VERSION}.nupkg" -o /tmp/roslyn.nupkg

# 3. Extract into the install directory and make executable
unzip -o /tmp/roslyn.nupkg -d ~/.local/share/roslyn
chmod +x ~/.local/share/roslyn/content/LanguageServer/linux-x64/Microsoft.CodeAnalysis.LanguageServer

# 4. Add to PATH in your shell rc file, then reload
echo 'export PATH="$HOME/.local/share/roslyn/content/LanguageServer/linux-x64:$PATH"' >> ~/.bashrc
source ~/.bashrc

# 5. Verify
Microsoft.CodeAnalysis.LanguageServer --version
----

[TIP]
====
Browse all versions at `${FEED}/index.json`. The `nuget.org` package of
the same name carries only a single stale prerelease — do not use it.
To pin a specific version instead of auto-resolving the newest, set
`VERSION=` manually (e.g. `VERSION=5.4.0-2.26179.14`).
====

=== Installing .NET SDK

Follow the official guide for Ubuntu / WSL:

[source,sh]
----
wget https://packages.microsoft.com/config/ubuntu/$(lsb_release -rs)/packages-microsoft-prod.deb
sudo dpkg -i packages-microsoft-prod.deb
sudo apt update && sudo apt install -y dotnet-sdk-8.0
----

Or download directly from https://dotnet.microsoft.com/download. `net10.0` (LTS to 2028) works
equally well — swap the package for `dotnet-sdk-10.0`.

[NOTE]
====
Debugging and running a project needs the **runtime that matches its `<TargetFramework>`**, not
just any SDK. A net10 SDK builds a `net8.0` project, but running/debugging it also needs the
net8.0 *runtime* (`dotnet --list-runtimes` should show `Microsoft.NETCore.App 8.0.x`; install with
`sudo apt install dotnet-runtime-8.0`). The whole toolchain here (roslyn.nvim, fsautocomplete,
csharprepl, easy-dotnet, netcoredbg) is SDK-version-agnostic — pick net8.0 or net10.0 and keep the
project target and installed runtime in sync.
====
