Common Lisp has a two-level namespace system.
*Packages* organize symbols at the language level: every symbol belongs to exactly one package, and `defpackage` controls what is exported and what is imported.
*Systems* (managed by ASDF) organize files on disk: an `.asd` file declares the compilation order, dependencies, and test hooks for a project.
*Quicklisp* is the de-facto package manager, fetching and installing ASDF systems from the internet.

Understanding all three is essential before building anything larger than a scratch file.

Open a scratch file and connect:

[source,sh]
----
nvim scratch.lisp
----

,cc

== 08 — Packages, Systems & Quicklisp

=== 1. The Default Package

When SBCL starts, your REPL is in the `COMMON-LISP-USER` package:

[source,lisp]
----
*package*
; => #<PACKAGE "COMMON-LISP-USER">

(package-name *package*)
; => "COMMON-LISP-USER"
----

`COMMON-LISP-USER` (nicknamed `CL-USER`) uses the `COMMON-LISP` package, which exports all ~978 standard symbols:
`defun`, `car`, `format`, `loop`, and so on. You get them all without qualification.

[source,lisp]
----
;; These are equivalent in CL-USER:
(car '(1 2 3))
(cl:car '(1 2 3))
; => 1
----

In your own packages you will control exactly what is visible.

'''

=== 2. defpackage — Declaring a Package

`defpackage` declares a package: what standard or third-party symbols it imports, what it exports for others to use, and optional nicknames.

[source,lisp]
----
(defpackage :my-app
  (:use :cl)                           ; import all CL standard symbols
  (:export #:greet #:farewell)         ; symbols others can use
  (:nicknames :app))                   ; my-app and app both refer to this package
----

Common options:

[cols="2,5",options="header"]
|===
| Option | Effect

| `:use :cl`
| Import the full ANSI Common Lisp standard namespace; almost always included

| `:export #:symbol`
| Make `symbol` accessible from other packages without qualification

| `:import-from :other-pkg #:fn1 #:fn2`
| Import specific symbols from `other-pkg` into this package's namespace

| `:nicknames :short`
| Additional names for the package; `short:foo` and `my-package:foo` are identical

| `:shadow #:symbol`
| Create a new `symbol` local to this package, hiding the inherited one

| `:shadowing-import-from`
| Import a symbol that conflicts with an inherited one, suppressing the conflict
|===

[source,lisp]
----
;; A realistic package with selective import
(defpackage :my-app.utils
  (:use :cl)
  (:import-from :alexandria
                #:flatten
                #:curry
                #:compose)
  (:export #:flatten-and-sum
           #:make-adder))
----

(Alexandria is not loaded in this scratch session — the form demonstrates the syntax.)

'''

=== 3. in-package — Switching the Current Package

After declaring a package you must enter it. `in-package` sets `*package*` for the rest of the file:

[source,lisp]
----
(defpackage :my-app
  (:use :cl)
  (:export #:greet))

(in-package :my-app)

;; Now we are inside my-app
(defun greet (name)
  (format nil "Hello, ~A!" name))

(greet "World")
; => "Hello, World!"
----

*Try it:* evaluate the `defpackage`, then `in-package`, then define `greet`, then call it — all with `,ee`.
Check `*package*` after `in-package`:

[source,lisp]
----
*package*
; => #<PACKAGE "MY-APP">
----

Switch back to `CL-USER` at any time:

[source,lisp]
----
(in-package :cl-user)
----

'''

=== 4. Qualified Names

When a symbol is *not* imported into the current package, you address it with a qualified name using the `::` or `:` separator:

[source,lisp]
----
;; One colon — access an exported symbol from outside the package
my-app:greet

;; Two colons — access any symbol (exported or not) from outside
;; Use sparingly — reaching into private symbols is a code smell
my-app::some-internal-helper
----

[source,lisp]
----
;; Demonstration — call greet from CL-USER without importing it
(in-package :cl-user)
(my-app:greet "Lisp")
; => "Hello, Lisp!"
----

[source,lisp]
----
;; The cl: prefix explicitly names a standard symbol
(cl:format cl:t "~&Pi is ~,4F~%" cl:pi)
; prints: Pi is 3.1416
----

'''

=== 5. ASDF — Another System Definition Facility

ASDF is the build system for Common Lisp. It ships with SBCL (and most other implementations) and is
always available in the container:

[source,lisp]
----
(require :asdf)
(asdf:asdf-version)
; => "3.3.x"   (exact version depends on SBCL version)
----

An `.asd` file sits at the root of a project and describes:

* the system name (a keyword)
* its source files and their load order
* its dependencies on other systems
* optional test operation configuration

.Anatomy of a minimal `.asd` file
[source,lisp]
----
;;; greeter.asd — placed at the root of the project

(asdf:defsystem :greeter
  :description "A simple CLI greeter"
  :version "0.1.0"
  :author "Your Name"
  :license "MIT"

  :depends-on (:alexandria)        ; other ASDF systems this one requires

  :serial t                        ; load components in order (top to bottom)
  :components ((:file "src/packages")
               (:file "src/greet")
               (:file "src/main"))

  :in-order-to ((asdf:test-op
                 (asdf:load-op :greeter/tests))))
----

`:serial t` is the safe default — it ensures each file is compiled and loaded before the next.
Without it you must declare `:depends-on` between individual components.

'''

=== 6. Project Layout

A conventional ASDF project looks like this:

----
greeter/
  greeter.asd          ← system definition
  src/
    packages.lisp      ← all defpackage forms
    greet.lisp         ← core logic
    main.lisp          ← entry point (main function)
  t/
    greeter-tests.asd  ← separate test system (or a subsystem)
    suite.lisp         ← test definitions
  README.md
----

Keeping `defpackage` forms in a dedicated `packages.lisp` (loaded first via `:serial t`) prevents circular
dependency issues: every other file can simply call `(in-package :my-package)` knowing the package exists.

[source,lisp]
----
;;; src/packages.lisp
(defpackage :greeter
  (:use :cl)
  (:export #:greet #:main))

;;; src/greet.lisp
(in-package :greeter)

(defun greet (name)
  (format nil "Hello, ~A!" name))

;;; src/main.lisp
(in-package :greeter)

(defun main ()
  (let ((name (or (second sb-ext:*posix-argv*) "World")))
    (format t "~A~%" (greet name))))
----

'''

=== 7. Quicklisp — Installing Third-Party Libraries

Quicklisp is pre-installed in the container. It provides `ql:quickload` to download and load a library
from the Quicklisp distribution:

[source,lisp]
----
;; Load a library (downloads it if not already cached)
(ql:quickload :alexandria)
; => (:ALEXANDRIA)

;; Load multiple libraries at once
(ql:quickload '(:alexandria :cl-ppcre))
; => (:ALEXANDRIA :CL-PPCRE)

;; Search for a library by keyword
(ql:system-apropos "json")
; Lists all Quicklisp systems whose name contains "json"

;; Update all installed distributions to the latest snapshot
(ql:update-all-dists)
----

After `ql:quickload` the library is available in the current session. Its symbols are accessible via
qualified names or by importing them into your package.

[source,lisp]
----
;; Alexandria: functional utilities
(ql:quickload :alexandria)

(alexandria:flatten '(1 (2 3) (4 (5 6))))
; => (1 2 3 4 5 6)

(alexandria:curry #'+ 10)
; => #<CLOSURE ...>

(funcall (alexandria:curry #'+ 10) 5)
; => 15
----

'''

=== 8. Loading Your Own System

To load a project you created, ASDF needs to find its `.asd` file.
The easiest approach in the container is to push the project directory onto the source registry:

[source,lisp]
----
;; Tell ASDF where to find your project (one-time per session)
(pushnew #p"/app/greeter/" asdf:*central-registry* :test #'equal)

;; Now load the system
(asdf:load-system :greeter)

;; Or use Quicklisp's load (tries Quicklisp first, falls back to ASDF)
(ql:quickload :greeter)
----

The container maps your working directory into `/app` (via the `LISP_DIR` bind mount from lesson 01).
If your project directory is in `$LISP_DIR`, it is already available at `/app` inside the container.

[source,lisp]
----
;; Verify your system loaded
(asdf:find-system :greeter)
; => #<ASDF/SYSTEM:SYSTEM "greeter" ...>
----

=== 9. Worked Example — A Two-File Project

Create these files in a real directory to try the full load cycle.
Assume the project lives at `~/projects/greeter/` and the container is started with `LISP_DIR=~/projects/greeter`.

.greeter/greeter.asd
[source,lisp]
----
(asdf:defsystem :greeter
  :description "A minimal greeter"
  :version "0.1.0"
  :serial t
  :components ((:file "src/packages")
               (:file "src/greeter")))
----

.greeter/src/packages.lisp
[source,lisp]
----
(defpackage :greeter
  (:use :cl)
  (:export #:greet #:greet-many))
----

.greeter/src/greeter.lisp
[source,lisp]
----
(in-package :greeter)

(defun greet (name)
  "Return a greeting string for NAME."
  (format nil "Hello, ~A!" name))

(defun greet-many (&rest names)
  "Greet each name in NAMES, one per line."
  (dolist (name names)
    (format t "~A~%" (greet name))))
----

In `scratch.lisp`, load and exercise the system:

[source,lisp]
----
;; Register the source directory (the container maps LISP_DIR to /app)
(pushnew #p"/app/" asdf:*central-registry* :test #'equal)

(asdf:load-system :greeter)
; Compiles and loads src/packages.lisp then src/greeter.lisp

(greeter:greet "Alice")
; => "Hello, Alice!"

(greeter:greet-many "Alice" "Bob" "Carol")
; prints:
;   Hello, Alice!
;   Hello, Bob!
;   Hello, Carol!
; => NIL
----

*Try it:* create the directory structure, start the container with the matching `LISP_DIR`, then
evaluate the three load forms above with `,ee`.
