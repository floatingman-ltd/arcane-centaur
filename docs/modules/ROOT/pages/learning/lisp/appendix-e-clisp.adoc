== Appendix E — CLISP

=== Why CLISP Is Not Containerized

The `docker/lisp-swank/` setup uses a single `swank-start.lisp` to start the Swank server across all implementations.
That script calls:

[source,lisp]
----
(swank:create-server :port 4005
                     :style :spawn
                     :dont-close t
                     :interface "0.0.0.0")
----

The `:style :spawn` option tells Swank to handle each connection in a *new thread*.
This requires native OS threads — the implementation must support `bordeaux-threads` or an equivalent.

SBCL, CCL, ECL, and ABCL all support native threads.
*CLISP does not.* It is a bytecode interpreter with no native thread support.
When CLISP tries to start Swank with `:style :spawn`, it signals an error at startup.

This is a fundamental limitation of CLISP's architecture, not a configuration gap.
Adding a CLISP profile would require a separate, single-threaded Swank startup that uses `:style nil` or `:style :sigio`.

'''

=== Running CLISP Locally

If you have CLISP installed on your host or in a custom environment and want to experiment, you can start Swank without threading:

First, install Quicklisp in your local CLISP:

[source,sh]
----
# Download Quicklisp
curl -O https://beta.quicklisp.org/quicklisp.lisp

# Bootstrap it under CLISP
clisp -norc -i quicklisp.lisp \
      -x '(quicklisp-quickstart:install)' \
      -x '(ql:add-to-init-file)' \
      -x '(quit)'
----

Then start a single-threaded Swank server:

[source,sh]
----
# Start Swank without threading (CLISP-compatible)
clisp -norc << 'EOF'
(load "~/quicklisp/setup.lisp")
(ql:quickload :swank)
(swank:create-server :port 4005
                     :style nil        ; no threads — single-threaded Swank
                     :dont-close t
                     :interface "127.0.0.1")
(loop (sleep 10))   ; keep the process alive
EOF
----

Alternatively, use `:style :sigio` for I/O-multiplexed handling (requires the host OS to support SIGIO):

[source,sh]
----
clisp -norc -x '(progn
  (load "~/quicklisp/setup.lisp")
  (ql:quickload :swank)
  (swank:create-server :port 4005 :style :sigio :dont-close t)
  (loop (sleep 10)))'
----

Connect Conjure as normal (`,cc` → 127.0.0.1:4005).

*Important:* with `:style nil` or `:style :sigio`, the Swank server is strictly single-threaded.
Concurrent evaluation requests will queue. Some Conjure features that rely on background threads
(e.g. the interrupt key) may not work as expected.

'''

=== CLISP Characteristics

* *Bytecode interpreter*: CLISP compiles Lisp to a portable bytecode, not native machine code.
  Performance is typically an order of magnitude below SBCL for compute-intensive code.
* *ANSI compliance*: CLISP has very good ANSI coverage. Code written strictly to the ANSI standard runs without modification.
* *Good for scripting*: CLISP starts faster than SBCL (no large image to load) and is a reasonable choice for Unix scripting tasks where you want Lisp semantics.
* *Small install*: the CLISP binary and its companion files total around 20–40 MB.
* *No native threads*: this is the defining limitation. Any library that requires `bordeaux-threads` or native concurrency will not work.
* *Largely dormant*: CLISP's last stable release is 2.49.93 (2010). Bug reports are rarely addressed. For new projects, SBCL or ECL are the recommended free implementations.

'''

=== Which Lessons Work on CLISP?

Lessons 01–08 use only ANSI Common Lisp forms and ASDF. They are portable and will run on CLISP with the single-threaded Swank workaround above.

Lesson 09 (building an application) uses `sb-ext:save-lisp-and-die` and the other implementation-specific save functions — none of those are CLISP's API. CLISP provides its own `(ext:saveinitmem "output.mem")` for saving images.

Lesson 10 (FiveAM) and Lesson 11 (Hunchentoot) depend on third-party libraries.
FiveAM works on CLISP. Hunchentoot requires native threads and will not start on CLISP.

Summary:

[cols="1,3,2",options="header"]
|===
| Lesson | CLISP compatible? | Note

| 01–08 | Yes | ANSI portable; Swank single-threaded workaround required
| 09 | Partially | `ext:saveinitmem` instead of `save-lisp-and-die`
| 10 | Yes | FiveAM is ANSI portable
| 11 | No | Hunchentoot requires native threads
|===

=== CLISP-Specific Save Function

For reference, if you do use CLISP locally:

[source,lisp]
----
;; Save a CLISP memory image
(pushnew #p"/your/project/" asdf:*central-registry* :test #'equal)
(asdf:load-system :greeter)

;; Save with an init function called at startup
(ext:saveinitmem "greeter.mem"
  :init-function #'greeter:main
  :quiet         t
  :norc          t)
----

Launch the saved image:

[source,sh]
----
clisp -M greeter.mem -- Alice
# Hello, Alice!
----

=== Recommendation

For new Common Lisp work, choose SBCL (fast, best supported), ECL (small/embeddable), or CCL (fast incremental, good macOS).
CLISP is useful for understanding historical Lisp or running ANSI-portable scripts on systems where only CLISP is available.
Its thread limitation makes it unsuitable for any concurrent or networked application.
