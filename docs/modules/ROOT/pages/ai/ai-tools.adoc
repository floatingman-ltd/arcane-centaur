== AI Tools Guide

This guide covers the AI / workflow tools that are integrated directly
into Neovim:

[width="100%",cols="50%,50%",options="header",]
|===
|Tool |What it does
|*Claude CLI* |Ask Claude to suggest shell commands or explain
code without leaving the editor

|*Research popup* |Ask a free-form question and read the answer in a
floating window — local (grounded in this config) or general knowledge

|*OpenSpec* |Drive the in-repo OpenSpec change workflow from Neovim
keymaps

|*Serena MCP server* |Provides symbol-aware code intelligence to AI
assistants such as Claude Code
|===

[[claude-cli]]
== Claude CLI

=== Usage

Select code in visual mode (or stay in normal mode to use the whole
buffer), then:

[width="100%",cols="50%,50%",options="header",]
|===
|Keys |Action
|`++<++leader++>++gcs` |Ask Claude to *suggest* a shell command based
on context

|`++<++leader++>++gce` |Ask Claude to *explain* the selected code
|===

The response appears in a floating window. Press `q` or `++<++Esc++>++`
to close it.

You can also run the commands directly: - `:ClaudeSuggest` — suggest -
`:ClaudeExplain` — explain

=== How it works

`lua/config/claude++_++cli.lua` uses `vim.system()` to invoke
`claude -p "++<++prompt++>++"` non-interactively, relying on Claude
Code's own authentication. The result is rendered in a scratch buffer
inside a `nvim++_++open++_++win` floating window.

'''''

=== Prerequisites

The `claude` CLI (Claude Code) must be on your `$PATH`. Install it from
https://claude.com/claude-code[claude.com/claude-code] and sign in once
(`claude` then follow the login prompt). No `ANTHROPIC_API_KEY` is
required for these commands — they use Claude Code's built-in auth.

[WARNING]
====
A stray `ANTHROPIC_API_KEY` in your environment *breaks* these commands. If the
variable (or another auth source) is set, the `claude` CLI uses it *instead of*
your Claude Code login, and the non-interactive `claude -p` path fails —
`:ClaudeSuggest` / `:ClaudeExplain` report:

`claude CLI failed (exit 1): ⚠ claude.ai connectors are disabled because ANTHROPIC_API_KEY … takes precedence over your claude.ai login`

The key is commonly exported from a shell init file such as `~/.zshenv`, which is
sourced by *every* shell — so Neovim inherits it and hands it to `claude -p`.

*Fix:* remove the export (e.g. from `~/.zshenv`) or `unset ANTHROPIC_API_KEY`. No
reboot is needed — `.zshenv` is re-read per shell. Confirm from a *fresh* terminal
(not a subshell; if using tmux, a new server or `tmux setenv -gu ANTHROPIC_API_KEY`):
`printenv ANTHROPIC_API_KEY` prints nothing, then a freshly launched Neovim runs the
commands without error.

Only the one-shot `claude -p` path is affected; the interactive *Claude Code Session*
(below) tolerates the variable.
====

''''''

== Claude Code Session (claudecode.nvim)

https://github.com/coder/claudecode.nvim[claudecode.nvim] provides a persistent,
editor-aware Claude Code session via the same WebSocket MCP protocol used by the
official VS Code and JetBrains extensions.

**How it differs from the one-shot CLI commands**

[cols=",,",options="header",]
|===
|Capability |Claude CLI (`<leader>gcs`/`gce`) |Session (`<leader>gcc`)
|Persistent session |No — exits after each command |Yes — Claude watches the session
|Sees open buffers/selection |No |Yes — via MCP context sharing
|Proposes diffs |No |Yes — native diff with accept/reject
|Requires ANTHROPIC_API_KEY |No |No — uses `claude` CLI auth
|===

Uses Neovim's native terminal provider (no snacks.nvim dependency).

=== Session Keymaps

[cols=",,",options="header",]
|===
|Keys |Mode |Action
|`++<++leader++>++gcc` |Normal |Toggle Claude Code session terminal
|`++<++leader++>++gcf` |Normal |Focus session terminal
|`++<++leader++>++gcb` |Normal |Add current buffer to Claude context
|`++<++leader++>++gcv` |Visual |Send selection to Claude
|`++<++leader++>++gca` |Normal |Accept Claude's proposed diff
|`++<++leader++>++gcr` |Normal |Reject Claude's proposed diff
|===

=== Prerequisites

The `claude` CLI must be on your `$PATH` (same requirement as the one-shot CLI commands).
No extra install needed — claudecode.nvim uses the same binary.

[[research-popup]]
== Research Popup

=== Usage

Press `++<++leader++>++?l` or `++<++leader++>++?a` in Normal mode. An input prompt appears — type your question and press Enter.

[width="100%",cols="34%,33%,33%",options="header",]
|===
|Keys |Command |Action
|`++<++leader++>++?l` |`:ResearchLocal` |Ask about *this* config — answer is grounded in your live keymaps and cheatsheets

|`++<++leader++>++?a` |`:ResearchAsk` |Ask a general-knowledge question — no local context
|===

The response appears in a floating window (same window as `:ClaudeExplain`). Press `q` or `++<++Esc++>++` to close it.

=== Local vs General

`:ResearchLocal` (`++<++leader++>++?l`) prepends your *live keymap descriptions* and *assembled cheatsheet content* to the prompt, then instructs the model to answer only from that context — it will say so when the answer is not present rather than guessing. Use this for config-specific questions like "how do I toggle the terminal?" or "what's the keymap for closing the diff view?".

`:ResearchAsk` (`++<++leader++>++?a`) sends your question verbatim with no local context. Use this for general programming and language questions.

=== How it works

`lua/config/research.lua` uses `vim.system()` to invoke `claude -p` asynchronously, relying on Claude Code's own authentication. The result is rendered in a scratch buffer inside a floating window via the shared `config.util.open_float` helper.

'''''

=== Prerequisites

The `claude` CLI (Claude Code) must be on your `$PATH`. No `ANTHROPIC_API_KEY` is required — these commands use Claude Code's built-in auth. See the xref:#claude-cli[Claude CLI prerequisites] section.

''''''

[[openspec]]
== OpenSpec

OpenSpec is the change-management workflow tool used in this repo (see
`openspec/` directory).

=== Usage

[width="100%",cols="34%,33%,33%",options="header",]
|===
|Keys |Command |Action
|`++<++leader++>++osn` |`:OpenspecNew` |Prompt for a change name and
create it

|`++<++leader++>++oss` |`:OpenspecStatus` |Show overall change status

|`++<++leader++>++osl` |`:OpenspecList` |List all changes
|===

*Scoped status:* pass a change name to `:OpenspecStatus` to see only
that change:

....
:OpenspecStatus add-ollama-learning-path
....

Output is shown in a read-only bottom split (15 lines tall). Press `q`
or `++<++Esc++>++` to close it.

=== How it works

`lua/config/openspec.lua` uses synchronous `vim.fn.system()`
(appropriate because `openspec` commands complete in well under a
second) and opens a scratch buffer in a `botright split`.

The same OpenSpec workflows are also available as Claude Code skills in
`.claude/skills/` — invoke them as slash commands (`/openspec-propose`,
`/openspec-apply-change`, `/openspec-continue-change`,
`/openspec-verify-change`, `/openspec-archive-change`).

'''''

=== Prerequisites

`openspec` must be on your `$PATH`. Install it following the project’s
own README, or check `openspec --help` to confirm it’s available.

[[serena]]
== Serena MCP Server

https://github.com/oramasearch/serena[Serena] is a symbol-aware
code-intelligence MCP (Model Context Protocol) server. When running, it
allows AI assistants such as Claude Code to perform semantic operations —
go-to-definition, find-references, workspace symbol search — that they
cannot do from text alone.

=== Configuration

Serena is configured for Claude Code at the user level in
`~/.claude.json` as a local MCP server, using the `serena` binary with
`serena start-mcp-server --context=claude-code --project-from-cwd`. With
`--project-from-cwd`, Serena picks up the `.serena/project.yml` in the
repo root automatically when Claude Code is launched from this directory.

=== Verifying

To confirm the MCP server is registered, run from a shell in the repo
root:

[source,sh]
----
claude mcp list
----

You should see a `serena` entry. Inside a Claude Code session you can
also confirm the `serena` tools are available.

'''''

=== Prerequisites

Install Serena so the `serena` binary is on `$PATH` (recommended via
`uv`):

[source,sh]
----
uv tool install serena-agent
----

[[avante]]
== AI Research — Avante

https://github.com/yetone/avante.nvim[avante.nvim] provides a Q&A chat
interface inside Neovim. It is configured *Ollama-only* — an offline, local
model with no external account or API key.

[NOTE]
====
The Claude (Anthropic) backend is intentionally *disabled*. Subscription
OAuth tokens are scoped by Anthropic's Terms of Service to Claude Code /
claude.ai, so driving them from a third-party tool risks a ToS violation.
See <<claude-disabled>> to re-enable it with an API key if you accept that
path.
====

[NOTE]
====
*Version:* avante.nvim is pinned to `v0.1.*` in `lua/plugins/avante.lua`
(prebuilt Linux binaries ship from `v0.0.29+`); the `build = "make"` step
downloads them. `dressing.nvim` has been removed — avante uses Neovim's
native `vim.ui` for its select/input prompts.
====

=== Keymaps (global)

[cols=",",options="header",]
|===
|Key |Action
|`++<++leader++>++aa` |Open avante with the current provider (ollama)
|`++<++leader++>++ao` |Select the ollama provider and open avante
|`++<++C++>++s` |Submit the prompt (inside the avante input; `++<++CR++>++` inserts a newline)
|===

=== Ollama Setup

Ollama runs as a persistent Docker service. *Both commands below are
required on first setup.*

*Step 1 — Start the container* (runs in the background; restarts
automatically on reboot):

[source,sh]
----
docker compose -f ~/.config/nvim/docker/ollama/docker-compose.yml up -d
----

*Step 2 — Download the model* (one-time per model; stored in a named
Docker volume so it survives container restarts). Avante's default is the
small `qwen2.5:0.5b` (~0.4 GB), chosen so it runs on very-limited-RAM machines.
Pull through Ollama's HTTP API rather than `docker exec` — `exec` sets up a
runc console socket that fails on some hosts with `failed to create runc
console socket: … read-only file system`; the API call talks straight to
the running server and avoids that:

[source,sh]
----
curl http://127.0.0.1:11434/api/pull -d '{"name":"qwen2.5:0.5b"}'
----

For more capability on machines with more RAM, pull a larger model
(e.g. `llama3.2:1b` ~1.3 GB, `llama3.2:3b` ~2 GB, or `llama3.1:8b` ~4 GB) and
set the matching tag as `model` in `lua/plugins/avante.lua`:

[source,lua]
----
model = "llama3.2:3b",
----

[NOTE]
====
Very small models (0.5–1B) tend to loop and repeat the final paragraph. The ollama provider sets
`repeat_penalty = 1.3` (above Ollama's `1.1` default) via `extra_request_body.options` in
`lua/plugins/avante.lua` to curb it. Raise it further, or move to a larger model, if looping
persists.
====

[TIP]
====
If the *container itself* won't start, fix Docker — this config keeps Ollama and
the other services containerized on purpose, so don't install them natively. A
fresh `docker run --rm alpine sh -c 'touch /t'` that fails with `read-only file
system` points at Docker's storage/overlay layer.
====

==== GPU Acceleration (NVIDIA)

GPU support is opt-in. Open `docker/ollama/docker-compose.yml` and
uncomment the `deploy` block:

[source,yaml]
----
deploy:
  resources:
    reservations:
      devices:
        - driver: nvidia
          count: 1
          capabilities: [gpu]
----

Requires the
https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/latest/install-guide.html[NVIDIA Container Toolkit].

==== When ollama is unreachable

If the ollama service is not running when you submit a question, avante
will display the HTTP error in its buffer. Neovim will not crash. Start
the service and retry.

[[claude-disabled]]
=== Claude Backend (disabled)

The Claude (Anthropic) provider is *not configured*. Two reasons:

* *Subscription OAuth* (`auth_type = "max"`) would let avante use a Claude
  Pro/Max plan without an API key — but Anthropic's Terms of Service scope
  those OAuth tokens to Claude Code / claude.ai, so using them from a
  third-party tool like avante risks a ToS violation.
* The API-key path (`ANTHROPIC_API_KEY`) avoids that, but adds an external
  account/billing dependency this config would rather not require.

So avante stays Ollama-only. To re-enable Claude with an API key (the
ToS-safe path), add a `claude` provider back to `lua/plugins/avante.lua` and
a `++<++leader++>++ac` mapping:

[source,lua]
----
providers = {
  ollama = { --[[ ... ]] },
  claude = {
    endpoint = "https://api.anthropic.com",
    model = "claude-3-5-haiku-20241022",
    auth_type = "api",
    api_key_name = "ANTHROPIC_API_KEY",
  },
},
----

Then `export ANTHROPIC_API_KEY='…'` (from https://platform.claude.com/) and
map `++<++leader++>++ac` to `require("avante.api").switch_provider("claude")`.
