.NET Guide (C# and F#)

Full support for C# and F# including LSP, REPL-driven development, and format-on-save. Both languages share the .NET SDK and the same iron.nvim REPL keymaps.

Quick Start

C#

  1. Open any .cs file — Roslyn attaches automatically.

  2. If the project has multiple .sln files, run :Roslyn target to pick the solution.

  3. Start the REPL with ,sl on any line, or open it explicitly via iron.nvim’s :IronRepl.

F#

  1. Verify prerequisites:

    dotnet --version
dotnet tool list -g | grep fsautocomplete

  2. Open a .fs source file — plugins and LSP attach automatically.

  3. Start the REPL with ,sl on any line.

Debugging (nvim-dap + netcoredbg)

Breakpoint debugging for C# and F# is provided by nvim-dap
nvim-dap-ui with the netcoredbg adapter, which easy-dotnet auto-registers with nvim-dap once both are loaded (there is no hand-written dap.adapters entry — a session therefore only starts from a .cs/.fsharp buffer, where easy-dotnet is loaded).

Prerequisites

netcoredbg is not a NuGet / dotnet tool package (there is no dotnet tool install -g netcoredbg). Download the prebuilt binary from the Samsung/netcoredbg releases and put it on your $PATH.

Linux / WSL2 (x64):

# Resolve the latest release tag, download the linux-amd64 build, and extract it
NCDBG_VER=$(curl -s https://api.github.com/repos/Samsung/netcoredbg/releases/latest \
  | grep '"tag_name"' | cut -d'"' -f4)
curl -L "https://github.com/Samsung/netcoredbg/releases/download/${NCDBG_VER}/netcoredbg-linux-amd64.tar.gz" \
  -o /tmp/netcoredbg.tar.gz
mkdir -p ~/.local/share/netcoredbg
tar -xzf /tmp/netcoredbg.tar.gz -C ~/.local/share/netcoredbg/

# Add to your shell rc (~/.bashrc or ~/.zshrc), then reload:
echo 'export PATH="$PATH:$HOME/.local/share/netcoredbg"' >> ~/.bashrc
source ~/.bashrc

Verify it is on your $PATH:

netcoredbg --version

If netcoredbg --version is not found afterward, the tarball extracted into a nested netcoredbg/ subdirectory — point PATH at ~/.local/share/netcoredbg/netcoredbg instead.

Every action has both a function-key and a <leader>b binding. The F-keys are the universal debugger idiom and work on server/SSH terminals; the <leader>b mirrors are for GUI terminal emulators that intercept function keys before Neovim sees them (commonly <F11> for fullscreen, <F10> for a menu). Both are always active — use whichever your terminal passes through.

The debug UI opens automatically when a session starts and closes on exit. Launch configurations are auto-discovered by easy-dotnet from the project structure (no hand-authored launch.json required).

Testing (easy-dotnet)

easy-dotnet.nvim provides a .NET test runner, project picker, and solution management using fzf-lua. It does NOT manage the Roslyn LSP — roslyn.nvim remains the sole C# language server.

easy-dotnet.nvim is a thin client over a separate EasyDotnet server tool, which powers all its features (debug, test, run, build). Install it once:

dotnet tool install -g EasyDotnet     # provides the `dotnet-easydotnet` executable
dotnet-easydotnet -v                  # verify (needs ~/.dotnet/tools on PATH)

Without it, every :Dotnet … action (and <F5> debug) errors 'dotnet-easydotnet' is not executable. Run :checkhealth easy-dotnet to confirm the server and the other dependencies (dap, fzf-lua, the c_sharp parser).

Test / Run / Build Keymaps (C# and F# buffers)

Keys Action

,tt

Run tests for selected project

,tr

Run selected project

,tb

Build selected project

Run/test output opens in a managed terminal that stays open on exit (we set managed_terminal.auto_hide = false; the default closes it the moment a run exits 0). Dismiss it with q.

easy-dotnet resolves the target project from Neovim’s current working directory (vim.fn.getcwd()), not the current buffer, and it caches one active project (shown in lualine). Open Neovim from the project’s directory (or :lcd %:p:h) so ,tr/,tt/,tb act on the project you mean.

F# projects need a solution file. easy-dotnet’s runnable-project discovery is C#-oriented, so a standalone .fsproj isn’t found ("No runnable projects found") even though dotnet run works — add a .sln (or .slnx) alongside it. C# .csproj projects are discovered without one.

Typical Workflow

 +------------------------------------------------------+
 |  Neovim                                              |
 |  +--------------------------+---------------------+  |
 |  |  Program.cs / hello.fs   |  REPL               |  |
 |  |                          |                     |  |
 |  |  var x = 42;             |  > var x = 42;      |  |
 |  |  Console.WriteLine(x);   |  42                 |  |
 |  |                          |                     |  |
 |  |  ,sl to send line -------+                     |  |
 |  +--------------------------+---------------------+  |
 +------------------------------------------------------+

  1. Open a .cs or .fs file — LSP starts and diagnostics appear.

  2. Send snippets to the REPL with ,sl / ,sc to explore APIs interactively.

  3. <leader>f (or save) formats with CSharpier / fsautocomplete.

  4. Use gd / gr / <leader>ca for full IDE-style navigation and refactoring.

Troubleshooting

Roslyn does not attach

Check that Microsoft.CodeAnalysis.LanguageServer is on your $PATH:

which Microsoft.CodeAnalysis.LanguageServer

If not found, re-run the install steps in Prerequisites.

fsautocomplete does not attach

Check that it’s installed:

dotnet tool list -g | grep fsautocomplete

Run :lua =vim.lsp.get_clients({bufnr=0}) inside Neovim to check whether the server is attached to the current buffer. To open the LSP log:

:lua vim.cmd('e ' .. vim.lsp.get_log_path())

:checkhealth lsp gives a broader overview of LSP status.

REPL does not open

Check that the REPL backend binary is on your $PATH:

which csharprepl      # C#
dotnet fsi --version  # F#

Installing .NET Global Tools

Installing the Roslyn Language Server

The Roslyn LSP is the official Microsoft C# language server used in VS Code’s C# Dev Kit. It is not installed via dotnet tool install, and it is not on the public nuget.org feed. It is published to Microsoft’s Azure DevOps vs-impl feed, and every release is a prerelease (there is no stable 5.x — the newest is 5.4.0-2.26179.14 at the time of writing; there is no 5.5/5.6).

Linux / WSL2 (x64):

# 1. Resolve the newest version from the Azure DevOps vs-impl feed
PKG=microsoft.codeanalysis.languageserver.linux-x64
FEED="https://pkgs.dev.azure.com/azure-public/vside/_packaging/vs-impl/nuget/v3/flat2/${PKG}"
VERSION=$(curl -s "${FEED}/index.json" | tr ',' '\n' | grep -oE '[0-9][0-9.]+-[0-9.]+' | sort -V | tail -1)
echo "Installing Roslyn LSP ${VERSION}"

# 2. Download the linux-x64 package (self-contained native binary)
mkdir -p ~/.local/share/roslyn
curl -L "${FEED}/${VERSION}/${PKG}.${VERSION}.nupkg" -o /tmp/roslyn.nupkg

# 3. Extract into the install directory and make executable
unzip -o /tmp/roslyn.nupkg -d ~/.local/share/roslyn
chmod +x ~/.local/share/roslyn/content/LanguageServer/linux-x64/Microsoft.CodeAnalysis.LanguageServer

# 4. Add to PATH in your shell rc file, then reload
echo 'export PATH="$HOME/.local/share/roslyn/content/LanguageServer/linux-x64:$PATH"' >> ~/.bashrc
source ~/.bashrc

# 5. Verify
Microsoft.CodeAnalysis.LanguageServer --version

Browse all versions at ${FEED}/index.json. The nuget.org package of the same name carries only a single stale prerelease — do not use it. To pin a specific version instead of auto-resolving the newest, set VERSION= manually (e.g. VERSION=5.4.0-2.26179.14).

Installing .NET SDK

Follow the official guide for Ubuntu / WSL:

wget https://packages.microsoft.com/config/ubuntu/$(lsb_release -rs)/packages-microsoft-prod.deb
sudo dpkg -i packages-microsoft-prod.deb
sudo apt update && sudo apt install -y dotnet-sdk-8.0

Or download directly from https://dotnet.microsoft.com/download. net10.0 (LTS to 2028) works equally well — swap the package for dotnet-sdk-10.0.

Debugging and running a project needs the runtime that matches its <TargetFramework>, not just any SDK. A net10 SDK builds a net8.0 project, but running/debugging it also needs the net8.0 runtime (dotnet --list-runtimes should show Microsoft.NETCore.App 8.0.x; install with sudo apt install dotnet-runtime-8.0). The whole toolchain here (roslyn.nvim, fsautocomplete, csharprepl, easy-dotnet, netcoredbg) is SDK-version-agnostic — pick net8.0 or net10.0 and keep the project target and installed runtime in sync.
